## When to Use Each

### pip (Traditional)
**Use when:**
- Building simple scripts
- Working in a legacy codebase
- Need minimal tooling
- No reproducibility requirements

**Example:**
```bash
python -m venv .venv
source .venv/bin/activate
pip install requests pandas
```

**Pros:**
- Minimal dependencies
- Everyone knows it
- Works everywhere

**Cons:**
- No lock file
- Slow resolution
- No dev dependencies
- Manual venv management

---

### Poetry
**Use when:**
- Publishing Python libraries/packages
- Need deterministic builds
- Want a single tool for package management
- Working on established Poetry projects

**Example:**
```bash
poetry new my-project
poetry add requests
poetry install
```

**Pros:**
- Beautiful pyproject.toml interface
- Handles publishing well
- Good documentation
- Dev dependencies built-in

**Cons:**
- Slower than uv (30-60 seconds)
- Requires learning Poetry CLI
- Heavier resource usage
- Installation takes longer for new projects

---

### uv (Recommended for 2026+)
**Use when:**
- Starting new Python projects
- Need fast dependency resolution
- Want a modern, single tool
- Building FastAPI/LLM applications (recommended!)

**Example:**
```bash
uv init my-project
cd my-project
uv add requests
uv sync
```

**Pros:**
- âš¡ 10â€“100Ã— faster than pip/Poetry

- Automatic venv management
- Fast project creation
- Excellent for monorepos

**Cons:**
- Still relatively new (v0.x)
- Smaller ecosystem than pip
- Some edge cases may not be covered

## Migration Examples

### From pip to uv
```bash
# Old way (pip)
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# New way (uv)
uv sync
```

### From Poetry to uv
```bash
# Old way (Poetry)
poetry install

# Convert poetry.lock and pyproject.toml
uv pip compile pyproject.toml

# New way (uv)
uv sync
```

## Key Differences in Workflow

### Adding a Package

**pip:**
```bash
pip install requests
pip freeze > requirements.txt  # Manual
```

**Poetry:**
```bash
poetry add requests
# Automatically updates pyproject.toml and poetry.lock
```

**uv:**
```bash
uv add requests
# Automatically updates pyproject.toml and uv.lock
```

### Running Scripts

**pip/Poetry:**
```bash
source .venv/bin/activate
python script.py
```

**uv:**
```bash
uv run script.py  # No need to activate!
```

## pyproject.toml vs Requirements.txt

Modern Python projects use `pyproject.toml` (PEP 517/518/621):

```toml
[project]
name = "agentic-ai"
version = "0.1.0"
dependencies = [
    "fastapi>=0.104.0",
    "langchain>=0.1.0",
    "pydantic>=2.0.0",
]

[project.optional-dependencies]
dev = ["pytest>=7.0", "ruff>=0.2.0"]

[tool.uv]
python-version = "3.11"
```

This replaces:
- `requirements.txt`
- `requirements-dev.txt`
- `setup.py`
- `.python-version` files

## Lock Files

Both Poetry and uv use lock files for reproducible installations:

**poetry.lock:**
- Human-readable but verbose
- Generated by Poetry

**uv.lock:**
- More compact format
- Generated by uv
- Faster to parse

Lock files solve:
```python
# Without lock:
pip install requests  # Installs version 2.31.0 today
# Tomorrow: pip install requests  # Installs version 2.32.0!
# Your code breaks unexpectedly

# With lock:
uv sync  # Always installs exact same versions
```
